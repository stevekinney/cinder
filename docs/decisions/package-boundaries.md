# Package Boundaries: Finishing the Extraction

**Status:** Accepted 2026-07-20 ŌĆö no code moved yet; execution plan below.
**Date:** 2026-07-20

## Summary

`@lostgradient/chat` proved the sibling-package pattern. The same treatment is owed to the
markdown/editing family. The recommendation is **two new published packages plus a lazy-import
pass** ŌĆö not eight. Everything else that looks like a candidate fails the bar.

## Why extract at all

The motivation is not file size. It is that `@lostgradient/cinder` forces editor-only
dependencies onto every consumer.

```jsonc
// packages/components/package.json ŌĆö peerDependencies
"@milkdown/ctx", "@milkdown/kit", "@milkdown/prose",
"prosemirror-inputrules", "prosemirror-model",
"prosemirror-state", "prosemirror-view"
```

Install cinder for a `<Button>` and your package manager asks you to satisfy a ProseMirror
editor stack. Those seven peers exist for exactly **two** components.

The `dependencies` block tells a worse story. These are declared, installed by every consumer,
and imported by **zero** components:

| Dependency                                                                                                             | Real consumer                                                             |
| ---------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| `remark-math`, `remark-rehype`, `rehype-katex`, `rehype-sanitize`, `rehype-stringify`, `hast-util-sanitize`, `comlink` | `markdown/rendering` subpath only ŌĆö a downstream-app API, not a component |
| `zod`, `@modelcontextprotocol/sdk`                                                                                     | `src/cli/mcp.ts` only                                                     |
| `@shikijs/engine-oniguruma`, `@shikijs/langs`, `@shikijs/types`                                                        | **Load-bearing, but declared in the wrong package** ŌĆö see below           |

The `@shikijs/*` entries deserve a correction, because the obvious read is wrong. They are never
imported by name anywhere in `packages/components`, which makes them look like dead weight. They
are not. They are used deliberately in `packages/markdown/src/rendering/highlighter.ts`, which
builds a highlighter via `shiki/core` + per-language dynamic imports specifically to avoid
pulling shiki's full ~10 MB grammar bundle:

```ts
import { createOnigurumaEngine } from '@shikijs/engine-oniguruma';
import { createHighlighterCore } from 'shiki/core';
const grammars = { typescript: () => import('@shikijs/langs/typescript') /* ŌĆ” */ };
```

So this is a **misplaced dependency, not an unused one**: declared by `packages/components`,
consumed by `packages/markdown`. Deleting them breaks syntax highlighting at runtime while
everything still typechecks. Publishing `@lostgradient/markdown` fixes this correctly ŌĆö the
deps move with the code that actually uses them.

## The state of play

Half this work is already done, which is easy to miss.
`packages/components/src/{markdown,editor,diff,commentary}/` are **generated re-export shims**
over private workspace packages:

```ts
// packages/components/src/markdown/index.ts
// Generated by scripts/generate-exports.ts ŌĆö do not edit by hand.
// Re-exports the public surface of `@cinder/markdown` as a cinder sub-path.
export { diff, pipeline } from '@cinder/markdown';
```

The headless logic already lives in `@cinder/markdown`, `@cinder/diff`, `@cinder/editor`,
`@cinder/commentary` ŌĆö all private, all unpublished. Cinder re-exports them and therefore
carries their dependency weight. So there are two separable jobs:

1. **Publish the headless packages** and delete the shims. No components involved.
2. **Extract the components** that sit on top of them.

## The structural check

Extraction is only clean if the dependency arrow points one way: new package ŌåÆ cinder, never
cinder ŌåÆ new package. It does.

Markdown coupling is contained to exactly three components ŌĆö `review-editor`, `diff-viewer`,
`markdown-editor`. The apparent hits in `textarea`, `json-schema-editor`, and `code-block` are
`@related` / `@avoidWhen` doc comments, not imports.

And the three are genuinely one family, not three independents:

```ts
// review-editor/review-editor-impl.svelte
import MarkdownEditor from '../markdown-editor/markdown-editor.svelte';
import DiffViewer from '../diff-viewer/diff-viewer.svelte';
```

`review-editor` composes both. They share milkdown/prosemirror, `diff-match-patch`, the
markdown pipeline, and commentary. **That is one package, not three.**

Usefully, `source-diff-viewer` and `diff-statistics` parse unified-diff text with hand-written
parsers and touch none of it. They stay in core, which keeps the seam tight.

## Proposal

### 1. `@lostgradient/markdown` ŌĆö headless, no Svelte

Absorbs `@cinder/markdown` + `@cinder/diff`, **plus the headless half of `@cinder/editor`**
(`template-render.ts`, `template-placeholders.ts`, `sanitize-html.ts`). Pipeline, rendering,
diff, templating. Carries the `@shikijs/*` deps that today sit misfiled in cinder.

Why it can't fold into a neighbor: it has no Svelte dependency and a different audience ŌĆö
server-side rendering, build scripts, anything that wants the pipeline without a UI. Folding it
into a component package would force Svelte on non-Svelte consumers.

Why `@cinder/diff` folds _in_: it is 3 files and 16 lines of shim surface, and it exists
principally as a dependency of `@cinder/markdown`. A standalone package for it is pure
apparatus.

### 2. `@lostgradient/editor` ŌĆö the Svelte editing surface

Components: `markdown-editor`, `review-editor`, `diff-viewer`.
Absorbs `@cinder/commentary` and the **ProseMirror half** of `@cinder/editor` ŌĆö
`component-runtime.ts`, `attach.ts`, `commands.ts`, the milkdown plugin runtime, `clipboard.ts`.

**`@cinder/editor` splits; it does not fold.** The tempting move is to send all of it here
because "it's the editor package," but that is wrong by the same headless-vs-Svelte test used
above. Its templating trio is deliberately DOM-free ŌĆö `template-placeholders.ts` says so in its
header (_"DEP-582: No ProseMirror or DOM dependencies"_), its only value imports are
`@cinder/markdown/rendering` and a dependency-free local module, and its milkdown references are
`import type` only, so they erase at runtime. Folding it in wholesale would drag rehype/hast into
a package that requires Svelte and milkdown peers, to serve an API none of its components call ŌĆö
reproducing the "editor library that also has buttons" problem one level down.

Peers: `@lostgradient/cinder`, `@lostgradient/markdown`, `svelte`, milkdown, prosemirror.

Why it can't split further: `review-editor` composes the other two directly. Splitting produces
three packages that can never be installed independently.

### Net effect on cinder

Roughly two dozen dependencies leave the core package: 3 milkdown peers, 4 prosemirror peers,
5 remark, 3 rehype, 3 `@shikijs/*`, and `unified`, `unist-util-visit`, `hast-util-sanitize`,
`js-yaml`, `comlink`, `diff-match-patch`. Cinder stops being an editor library that also has
buttons.

`shiki` itself stays, because `code-block` stays. Worth noting while we are in here: the repo
runs **two different shiki strategies**, and cinder has the worse one.
`packages/markdown/src/rendering/highlighter.ts` deliberately uses `shiki/core` plus per-language
dynamic imports to avoid the full grammar bundle it measures at ~10 MB. Cinder's own
`src/highlighters/shiki/index.ts` does `await import('shiki')` and reaches for `bundledLanguages`
ŌĆö i.e. the whole thing. It is lazy, so it does not hit first paint, but any consumer who renders
one code block pays for every grammar. That is a pre-existing weight bug, independent of this
proposal, and it should be filed separately rather than absorbed here.

## What does _not_ become a package

Every remaining "candidate" fails the bar. Cinder already lazy-loads heavy deps in three
places ŌĆö `schema-form` defers `ajv`, `highlighters/shiki` defers `shiki`,
`_internal/anchored-overlay` defers `@floating-ui/dom`. That is the cheaper tool, and it is
already the house pattern.

| Dependency               | Components                      | Verdict                                                                                                           |
| ------------------------ | ------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `shiki`                  | code-block, approval-card       | Already dynamically imported. Move to an optional peer; no package                                                |
| `ajv`                    | schema-form, json-schema-editor | `schema-form` defers it; `json-schema-editor` imports it **statically**. Fix the inconsistency, not the packaging |
| `qrcode`                 | qr-code                         | One component. Lazy import                                                                                        |
| `libphonenumber-js`      | phone-input                     | One component, already on the `/min` subpath. Lazy import                                                         |
| `@tanstack/virtual-core` | tree, data-grid                 | Small, and virtualization is core infrastructure. Leave                                                           |
| `@floating-ui/dom`       | 10 components                   | Broadly used. Core. Not a candidate                                                                               |

Deliberately out of scope: charts, `data-grid`, `kanban-board`, `spectrogram`. They are large,
but size is not the bar ŌĆö none carries a heavy unique dependency or an optionality win.

Cheap wins available immediately, independent of any extraction:

- Make `json-schema-editor` defer `ajv` the way `schema-form` already does. Today it imports
  `ajv`, `ajv/dist/2019.js`, and `ajv/dist/2020.js` statically.
- `zod` and `@modelcontextprotocol/sdk` are CLI-only (`src/cli/mcp.ts`, reachable via `bin`) but
  ship to every consumer. Make them optional, or split the CLI.
- Do **not** drop `@shikijs/*` as dead weight ŌĆö see the correction above. They move to
  `@lostgradient/markdown` rather than being deleted.

## Cost per package

Derived from what `@lostgradient/chat` actually required. This is the counterweight that keeps
the count at two.

The toolchain is **shared, not cloned** ŌĆö sibling packages reach into
`../../components/scripts/` for the audit, publish, and generator logic. Chat carries ~10
scripts against components' ~90. Per-package cost is real but bounded:

- **Own scripts:** `build.ts`, `pack-for-publish.ts`, `validate-consumer.ts`,
  `generate-component-artifacts.ts`, `package-boundary.test.ts`
- **Own baselines:** `platform-viewport-baseline.json`, `raw-color-baseline.json`,
  `token-usage-baseline.json`, `coverage-ratchet.json`
- **Root `package.json`:** workspaces entry, `lint-staged` block, and `--filter` added to six
  scripts ŌĆö always _after_ cinder, since the peer contract targets a cinder version released in
  the same PR
- **Six workflow files:** `changeset-guard`, `browser-tests`, `deploy-playground`, `unit-tests`,
  `main-green`, `release` (+ `release-manual`)
- **Playground:** one `ComponentSource` entry in `src/component-sources.ts`
- **`packages/testing`:** nothing. It is fully generic

One trap worth naming: **chat's `exports` map is hand-authored, not generated.** Cinder has
`exports:generate`; chat does not. Budget for maintaining it by hand, guarded by
`package-boundary.test.ts`.

## Consequences

Deleting the shims shrinks cinder's public API. Consumers importing
`@lostgradient/cinder/markdown/*`, `/editor/*`, `/commentary/*`, `/diff/*` break and must
depend on the new packages. At `0.16.x` and pre-release this is acceptable and on-brand ŌĆö but
it is a real breaking change, and it should be a deliberate major/minor with a migration note,
not a surprise.

`@lostgradient/chat` is the first-party consumer that breaks, and it is sitting right here in the
repo. It dynamically imports `@lostgradient/cinder/markdown/rendering`, composes
`@lostgradient/cinder/code-block`, and `@import`s cinder style subpaths. Its peer set and CSS
imports must migrate to `@lostgradient/markdown` in the same release train ŌĆö it is the migration
rehearsal for every external consumer, so do it first and honestly.

Sequencing: publish `@lostgradient/markdown` first (it has no Svelte dependency and no
component churn), then `@lostgradient/editor` against it. Cinder's shims come out last, in
the same release train.

## Decision

**Accepted 2026-07-20: publish, two packages.** Keeping them as internal boundaries was the
alternative and it was rejected ŌĆö it leaves cinder carrying the weight forever, which is the
entire problem this solves.

### Naming: there is only one published scope

`@lostgradient` is the only npm scope available, and it currently holds `cinder`, `chat`, and
`weft`. `@lostgradient/markdown` is unregistered, so both target names are free.

The `@cinder/*` packages therefore do **not** "graduate" ŌĆö nothing about that name carries over.
They are workspace-internal identifiers that have never touched the registry, and anything
published gets a new `@lostgradient/*` name. Concretely, every `@cinder/*` package involved here
stops existing:

| Today                | After                                              |
| -------------------- | -------------------------------------------------- |
| `@cinder/markdown`   | renamed ŌåÆ **`@lostgradient/markdown`** (published) |
| `@cinder/diff`       | absorbed into `@lostgradient/markdown` ŌĆö gone      |
| `@cinder/editor`     | split across both new packages ŌĆö gone              |
| `@cinder/commentary` | absorbed into `@lostgradient/editor` ŌĆö gone        |
| `@cinder/playground` | unchanged, stays private                           |
| `@cinder/testing`    | unchanged, stays private                           |

That leaves a rule worth keeping: **`@cinder/*` means private dev tooling that will never ship;
`@lostgradient/*` means published.** Today the two scopes are muddled ŌĆö `@cinder/markdown` is
private but its code ships inside cinder's bundle anyway. Afterwards the scope tells you the
truth.

Every `workspace:*` reference to a renamed package must be repointed: `packages/components`,
`packages/playground`, and the internal edges between `commentary`, `editor`, and `markdown`.

One collision the naming creates: **`packages/editor/` is already occupied** by the private
`@cinder/editor`, and `packages/markdown/` by `@cinder/markdown`. Both directories are wanted by
the new published packages under the same short names. The sequencing below avoids a rename
dance by emptying each directory before reusing it ŌĆö `@cinder/editor` is fully dissolved in Phase
1, so `packages/editor/` is free by the time Phase 3 wants it. Do not create a
`packages/editor-new/` or similar; a transitional directory name is exactly the kind of
compatibility scaffolding that outlives its purpose.

Also note the collision with cinder's own `src/editor/` shim directory. That is unrelated code ŌĆö
a generated re-export barrel ŌĆö and it is deleted in Phase 5 regardless.

## Execution plan

Six phases, each its own pull request. The ordering is load-bearing: a package can only be
depended on after it is published, and cinder's shims can only come out once every first-party
consumer has moved off them.

**Phase 0 ŌĆö cheap wins, no boundary changes.** Make `json-schema-editor` defer `ajv`. Make `zod`
and `@modelcontextprotocol/sdk` optional (or split the CLI). File the `code-block` full-bundle
shiki issue separately. Independently shippable; touches no package boundary and unblocks
nothing, so it can land in parallel with Phase 1.

**Phase 1 ŌĆö carve `@cinder/editor` in two.** Move `template-render.ts`,
`template-placeholders.ts`, `sanitize-html.ts`, and `placeholder-security.ts` into
`@cinder/markdown`. Move the ProseMirror remainder ŌĆö `component-runtime.ts`, `attach.ts`,
`commands.ts`, the milkdown plugin runtime, `clipboard.ts` ŌĆö into `@cinder/commentary`, which is
already milkdown-coupled and is the thing Phase 3 turns into `@lostgradient/editor`. That fully
dissolves `@cinder/editor` and frees `packages/editor/`. Still private, still internal ŌĆö pure
code movement inside the workspace, verifiable by `bun run validate`, with no publishing risk.
Do it before anything is published so the first published surface is already the right shape.

**Phase 2 ŌĆö publish `@lostgradient/markdown`.** Rename `@cinder/markdown`, absorb `@cinder/diff`,
and repoint every `workspace:*` reference. Move the `@shikijs/*` deps off `packages/components`
and onto this package. No Svelte dependency, no component churn.
Follow the chat checklist: hand-authored `exports` map, own `build.ts` / `pack-for-publish.ts` /
`validate-consumer.ts`, own baselines, root `package.json` wiring, six workflow files. Gate on
`validate:consumer` before the first real publish.

**Phase 3 ŌĆö publish `@lostgradient/editor`.** Components `markdown-editor`, `review-editor`,
`diff-viewer`, plus `@cinder/commentary` and the ProseMirror half of `@cinder/editor`. Peers:
cinder, `@lostgradient/markdown`, svelte, milkdown, prosemirror. Needs a playground
`ComponentSource` entry and example fixtures per component, or artifact generation fails.

**Phase 4 ŌĆö migrate `@lostgradient/chat`.** Repoint its `markdown/rendering` import and CSS
`@import`s at `@lostgradient/markdown`. This is the migration rehearsal ŌĆö whatever hurts here
will hurt external consumers, so fix the ergonomics before writing the migration note.

**Phase 5 ŌĆö delete cinder's shims.** Remove `src/{markdown,editor,diff,commentary}/`, drop the
generated re-exports from `generate-exports.ts`, and strip the newly-orphaned dependencies and
milkdown/prosemirror peers. This is the breaking change and the payoff, and it ships with a
migration note mapping every removed subpath to its new home.

Watch items carried from the analysis: `exports-drift.test.ts` pins the CSS style-subpath shape
and will need updating; every new component family needs a seeded `README.md` and a hand-add to
`src/styles/components.css`; the release train publishes cinder first, then siblings.

## Delivery: waves and pull requests

Six pull requests in four waves. Wave 1 is fully parallel ŌĆö its three pull requests touch
disjoint files and share no dependency. Everything after is sequential because each wave
depends on the previous one being merged (and, for publishes, released).

| PR   | Branch                            | Contents                                           | Depends on |
| ---- | --------------------------------- | -------------------------------------------------- | ---------- |
| PR-A | `feat/turborepo-ci`               | Turborepo adoption + change-aware CI               | ŌĆö          |
| PR-B | `fix/dependency-hygiene`          | Phase 0 cheap wins                                 | ŌĆö          |
| PR-C | `refactor/dissolve-cinder-editor` | Phase 1: dissolve `@cinder/editor`                 | ŌĆö          |
| PR-D | `feat/publish-markdown`           | Phase 2: `@lostgradient/markdown` + release wiring | C          |
| PR-E | `feat/publish-editor`             | Phase 3: `@lostgradient/editor`                    | D released |
| PR-F | `feat/finish-extraction`          | Phases 4+5: chat migration + shim deletion         | E released |

Phases 4 and 5 share one pull request deliberately: both are halves of the same breaking
release train, and splitting them would ship an intermediate state where chat has migrated but
the shims it no longer needs still bloat cinder. PR-E must precede PR-F because the in-repo
`markdown-editor` component imports cinder's `./editor/component-runtime` shim until the
component moves.

## CI/CD and publishing changes

**`release.yaml`** grows from two packages to four. Publish order follows the peer DAG:
`@lostgradient/markdown` ŌåÆ `@lostgradient/cinder` ŌåÆ `@lostgradient/editor` ŌåÆ
`@lostgradient/chat` (markdown and cinder are mutually independent; editor peers on both;
chat peers on cinder and markdown). Each new package gets validate / weight-check / publish /
dry-run steps in that order, `changeset:version` gains the editor package's
`components:generate`, `changeset-guard.yaml` watches the new manifests, and
`release-manual.yaml` gains both packages as dispatch choices.

**First-publish bootstrap ŌĆö the one human step.** npm Trusted Publishing cannot mint a
brand-new package name; the package must exist before a Trusted Publisher can be configured
(this repo already relearned this once: the 0.2.0 ENEEDAUTH incident). So each new package
needs a one-time bootstrap: publish via `release-manual.yaml`'s token path (requires a current
`NPM_TOKEN` with permission to create packages in the `@lostgradient` scope), then configure
the Trusted Publisher for `release.yaml` on npmjs.com. Two packages, two bootstraps ŌĆö or do
both in the same sitting after PR-D and PR-E respectively.

## Turborepo and change-aware testing

**Decision: adopt, in its own pull request (PR-A), CI-first.**

The repo has hand-rolled the caching half of Turborepo already: `shouldSkipBuild` hash-skip
logic is replicated across five `build.ts` files, complete with comments managing the exact
mid-rebuild race conditions (`"a dependent must not compute its skip hash while its upstream is
mid-rebuild"`) that a task graph solves by construction. Root scripts and the playground `dev`
script hand-chain up to six `--filter` builds in dependency order. Going from four workspace
packages to a published DAG of four makes maintaining that by hand strictly worse.

Shape of the adoption:

- `turbo.json` models the graph once: `build` depends on `^build` with `dist/**` outputs;
  `test` and `typecheck` depend on `^build` (stale sibling `dist/` already breaks
  `svelte-check` today); `lint` is independent.
- Root scripts delegate to `turbo run`; per-package scripts are unchanged. `packageManager`
  field added (Turborepo requires it).
- **PR CI runs `--affected`** against `origin/main`, so a chat-only pull request stops
  building and testing cinder. `.turbo` is cached via `actions/cache`.
- **`main-green` stays a full run** ŌĆö no `--affected` ŌĆö as the safety net against
  affected-detection gaps. Publishes already gate on same-SHA main-green, so the release
  path's guarantees are unchanged.
- **Browser tests stay a full, ungated-by-affected suite.** They pin component class
  selectors and cross component boundaries; mapping changed files to affected Playwright
  specs is brittle. The playground's `component-filter.ts` slug allow-list is the future
  lever if suite time becomes a problem ŌĆö noted, not built.

Explicit non-goals for PR-A, so it stays reviewable: no remote cache (single-runner CI gains
nothing today), no husky/hook rewiring, no removal of `shouldSkipBuild` yet. The hash-skip
layer becomes redundant once turbo owns orchestration and gets deleted in a follow-up once
turbo has baked on main ŌĆö running both systems briefly is belt-and-suspenders, not drift, since
each is independently correct.
